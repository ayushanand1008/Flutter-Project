# Flutter E2EE Scrapbook: Backend & Architecture Blueprint (v7-Final)

## 1. Project Overview
A private, serverless, end-to-end encrypted mobile application for two users to sync and manage shared photographs. The app bypasses traditional cloud storage costs by leveraging the users' existing Google Drive storage, using Firebase exclusively for the initial account handshake and cryptographic key exchange. The app is designed exclusively for private distribution via sideloaded APK.

## 2. Build, Distribution & Authentication Strategy
Because this app bypasses the Google Play Store for direct private distribution, the build and authentication architecture must be strictly configured for a sideloaded environment.

* **Single Keystore & APK Build:** The app will be built as a single signed release APK (not an AAB/App Bundle). A dedicated release keystore must be generated. The SHA-1 and SHA-256 fingerprints from this exact keystore must be registered in the Google Cloud Console OAuth client and the Firebase project. Both partners must install this exact same signed APK for Google Sign-In to succeed.
* **Individual Authentication:** Despite sharing the same APK, both users will sign in individually using their own separate Google Accounts. There is no shared login. The Firebase Auth and Google Drive API integrations must instantiate using the local user's distinct credentials.
* **OAuth Consent & Unverified App Screen:** Because the app is not verified by Google, the OAuth consent screen will present a "Google hasn't verified this app" warning. Both users must be instructed to click **Advanced > Go to app (unsafe)** during onboarding. 
    * *Scope Implications:* Sticking to the `drive.file` scope keeps the app in a "sensitive" tier. Escalating to the full `drive` scope pushes it into a "restricted" tier. While both will show the unverified warning for up to 100 users, restricted scopes carry a higher risk of Google enforcing a strict block on the OAuth client without a formal CASA security assessment. This makes resolving the `.file` scope subfolder inheritance (Section 5) critical for long-term stability.
* **Play Store Exclusions:** The architecture must strictly exclude Play Store-specific implementations such as Play App Signing, dynamic feature modules, or the Play Integrity API.

## 3. Tech Stack & State Management
* **Framework:** Flutter
* **State Management:** `Provider` (for reactive UI updates and clear separation of logic)
* **Handshake / DB:** Firebase Firestore (Spark Plan - Free Tier)
* **Storage Backend:** Google Drive API (Evaluating `drive.file` vs `drive` scope)
* **Local Security:** Authenticated Encryption via **AES-256-GCM** with PBKDF2 key derivation. `local_auth` for session gating.

## 4. The Handshake, Salt Sync & State Rehydration (Firebase Firestore)
To ensure both devices derive the exact same Master Key from the shared password, the cryptographic salt is generated once during the pairing handshake. 

### **Zero-Knowledge Constraint**
The shared "couple password" is **NEVER** transmitted to Firebase, logged, or hashed server-side. It exists purely in local volatile memory for PBKDF2 derivation. 

### **Firestore Schema & Security Rules**
* **Collection:** `pairing_codes`
    * `code` (String): 6-digit random number.
    * `creator_uid` (String): Firebase Auth UID of User A.
    * `expires_at` (Timestamp): 15 minutes from creation.
* **Collection:** `couples`
    * `document_id` (String): Merged UID (e.g., `uidA_uidB`).
    * `user_a` (String): UID of User A.
    * `user_a_email` (String): Google account email of User A.
    * `user_b` (String): UID of User B.
    * `user_b_email` (String): Google account email of User B (populated upon handshake completion).
    * `crypto_salt` (String): A cryptographically secure random 16-byte salt generated by User A.
    * `master_drive_folder_id` (String): The ID of the shared Google Drive folder.
    * `handshake_status` (String): Enum-like state (`"pending_folder"`, `"ready"`, `"terminating"`).
    * `created_at` (Timestamp).
* **Security Rules Enforcement:** Firestore rules MUST restrict access to the `couples` document to only the specific authenticated UIDs listed in `user_a` and `user_b`.

### **State Rehydration (Post-Reinstall)**
On every sign-in, the app queries the `couples` collection for a document where `user_a` or `user_b` matches the signed-in UID. If found and `handshake_status` is `"ready"`, the app automatically restores local state (`master_drive_folder_id`, `crypto_salt`, partner email). The user is then prompted solely for the "couple password" to re-derive the encryption key in-memory.

    * *Drive Picker step eliminated:* Under full drive scope, User B's client has direct API access to all shared folders by ID. The one-time Picker step from the original onboarding flow is no longer required.
* **Onboarding Sequencing (Full `drive` Scope):**
    1. User A creates the master folder (e.g., `Our_Scrapbook`).
    2. User A retrieves `user_b_email` from the Firestore `couples` document.
    3. User A calls the Drive Permissions API to grant `writer` permissions to User B's email address for the master folder.
    4. User A writes the `master_drive_folder_id` to Firestore and sets `handshake_status` to `"ready"`.
    5. User B's UI detects `"ready"` status and directly accesses the shared folder by its stored ID using the full `drive` scope — **no Picker action required.**
* **Subfolder Naming Convention:** `YYYY-MM-DD_location_name` (e.g., `2025-09-15_Udaipur`).
    * *Sanitization Rule:* The text input for the location passes through a regex filter: `location.replaceAll(RegExp(r'[^a-zA-Z0-9\s\-_]'), '').trim().replaceAll(RegExp(r'\s+'), '_')`.
* **Subfolder Permissions API Call:** When a new subfolder is created, the creating app pulls the partner's email from the Firestore `couples` document (`user_a_email` or `user_b_email`) and explicitly calls the Drive Permissions API to share that specific subfolder.

## 6. Security Payload, Auth & Session Gating (AES-256-GCM)
* **Key Derivation:** Both devices derive the identical 256-bit symmetric Master Key using PBKDF2 with HMAC-SHA256, the shared couple password, and the `crypto_salt` retrieved from the Firestore `couples` document.
* **Session Lifecycle & App Lock Constraints:** The Master Key and the couple password are **never** cached to disk or secure storage. They exist exclusively in volatile memory for the duration of the OS process.
    * An in-app PIN/Biometric lock is implemented solely to gate access to the active, running session. Upon process death, the in-memory key is destroyed. On next launch, the user MUST manually re-enter the couple password to re-derive the Master Key.
* **Encryption/Decryption Flow:** Files are encrypted individually using AES-256-GCM with a unique 12-byte IV per file. The uploaded binary blob is packed as: `[12 bytes IV] + [16 bytes Auth Tag] + [Ciphertext]`.

## 7. Non-Destructive Exit Strategy (Bulk-Decryption Pipeline)
To eliminate the risk of data loss or corruption during account deletion, the exit strategy behaves as a safe, non-destructive migration, while ensuring the non-triggering partner is gracefully unpaired.
* **Trigger:** Protected "Export/Decrypt All Data" sequence in settings.
* **Safe Pipeline:**
    1. **Tombstone Flag:** The triggering app sets `handshake_status` to `"terminating"` in the Firestore `couples` document. The partner's app, listening to this stream, detects the flag, alerts the user, and safely clears its local state to prevent orphaned sync errors.
    2. App creates a distinct root directory in Google Drive titled `Our_Scrapbook_Decrypted_Export`.
    3. App initializes a worker queue to sequentially download, decrypt in-memory, and write raw bytes as `.jpg`/`.png` inside the new Export folder.
    4. **Cleanup Protocol:** Only after the entire vault is verified via Drive API read-readiness, the user is prompted to delete the original encrypted `Our_Scrapbook` directory, followed by a cascade cleanup of the Firestore document, OAuth tokens, and local device state.

6. **EXECUTE PRE-FLIGHT TEST:** Write a temporary script to test `drive.file` subfolder API inheritance between two test accounts. If successful, proceed with `drive.file`. If it fails (returns 403/404), switch the OAuth configuration to the full `drive` scope.

8. Apply the regex location string sanitization rule to the subfolder generation service. Ensure the Drive Permissions API explicitly shares every newly created subfolder with the partner's email.
9. Build the Safe Exit Strategy queue, including the "tombstone" status listener and the cleanup cascade.
